# 贡献指南

感谢您对 Linux 运维排错笔记项目的关注和支持！我们欢迎并鼓励社区成员贡献新的故障案例、改进现有内容或修复错误。

## 贡献方式

### 1. 提交新的故障案例

如果您遇到了一个值得分享的 Linux 运维故障案例，请按照以下步骤提交：

1. **选择合适的分类**：根据故障类型选择对应的目录（CPU、内存、磁盘、网络、服务）
2. **案例命名格式**：`案例X：简短描述.md`（例如：`案例1：生产服务器 CPU 100% 告警.md`）
3. **案例内容结构**：
   - 故障现象：详细描述故障发生时的现象
   - 排查步骤：逐步记录排查过程
   - 根因分析：分析故障的根本原因
   - 解决方案：提供具体的解决方法
   - 预防措施：如何避免类似故障再次发生
   - 参考命令：相关的 Linux 命令及其输出

### 2. 改进现有内容

如果您发现现有内容需要改进或存在错误，请：

1. 直接修改对应文件
2. 在提交时说明修改的原因和内容

### 3. 其他贡献

- 改进文档结构
- 补充排错工具使用说明
- 完善故障案例索引

## 内容规范

为了保持项目的一致性和质量，请遵循以下规范：

### 1. 文档格式

- 使用 Markdown 格式编写
- 标题层级清晰（#、##、### 等）
- 代码块使用 ```bash 或 ``` 标记
- 命令输出使用代码块展示
- 重要内容可以使用加粗或引用标记

### 2. 内容要求

- 故障描述要具体、真实
- 排查步骤要详细、可操作
- 根因分析要深入、准确
- 解决方案要可行、有效
- 预防措施要实用、有针对性

### 3. 语言规范

- 使用简体中文编写
- 语言表达要清晰、准确
- 避免使用过于专业的术语，必要时给出解释
- 保持一致的术语使用（如 CPU 使用率、内存泄漏等）

### 4. 命令示例

- 命令前使用 `$` 提示符
- 命令输出直接展示，不加提示符
- 敏感信息（如 IP 地址、密码等）请使用占位符

## 提交流程

1. **Fork 仓库**：在 GitHub 上 Fork 本项目到您的个人账户
2. **Clone 仓库**：将 Fork 后的仓库 Clone 到本地
3. **创建分支**：创建一个新的分支用于您的贡献
4. **提交更改**：在本地进行修改并提交
5. **创建 Pull Request**：向本项目提交 Pull Request，说明您的贡献内容

## 审核标准

我们会根据以下标准审核您的贡献：

- 内容的准确性和实用性
- 格式的规范性和一致性
- 案例的代表性和参考价值
- 描述的清晰度和完整性

## 联系我们

如果您在贡献过程中遇到任何问题，或有任何建议，请通过以下方式联系我们：

- GitHub Issues：在项目仓库中创建 Issue
